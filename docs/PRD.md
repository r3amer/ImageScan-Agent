## 1. 产品概述

### 1.1 产品定义

ImageScan-Agent 是一个基于 Google Gemini 2.5 Flash LLM 的 Docker 镜像敏感凭证扫描智能体系统。该系统采用主从 Agent 架构，通过 LLM 理解文件内容和上下文，智能识别容器镜像中的敏感凭证（API Keys、密码、证书等）。

### 1.2 核心价值

- **智能化**: 利用 LLM 理解文件内容和上下文，超越简单的正则表达式匹配
- **准确性**: 通过多 Agent 协作和置信度评估，降低误报率和漏报率
- **高度自主**: Agent 系统自主规划、执行、验证，仅需异常情况需人工介入

### 1.3 目标用户

- **主要用户**: 开源社区
- **使用场景**:
  - DevSecOps 流程中的镜像安全检查
  - CI/CD 流水线集成
  - 本地开发环境的快速扫描

### 1.4 成功指标

**对比基准**: 以超越 TruffleHog 等传统扫描工具为目标
- **并行对比**: 使用相同镜像测试 ImageScan-Agent 和 TruffleHog，比较结果

## 2. 功能规格

### 2.1 MVP 功能范围（最小可行产品）

#### 2.1.1 核心功能

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **镜像解压** | 保存 Docker 镜像为 tar 文件并解压，提取各层文件 | P0 |
| **智能文件筛选** | 使用 LLM 分析文件名，识别潜在敏感文件（如 .env、config.json） | P0 |
| **内容扫描** | 提取文件内容并扫描凭证，使用 LLM 进行语义分析 | P0 |
| **结果输出** | 输出 JSON 格式扫描报告，包含元数据、凭证列表、统计信息 | P0 |

#### 2.1.2 CLI 功能

```bash
imagescan agent scan <image>    # 扫描镜像
imagescan history               # 查看历史扫描记录
imagescan config                # 查看/修改配置
imagescan verify <credential>   # 验证凭证有效性
```

### 2.2 后续迭代功能

#### 2.2.1 Web UI（轻量级控制台）
- 查看扫描结果（类型分类、脱敏展示、位置信息、置信度）
- 简单的扫描触发按钮

#### 2.2.2 高级 Agent 功能
- 凭证有效性验证（静态分析）
- RAG 知识库集成（历史案例库、凭证模式库）
- 外部知识库集成（CVE、安全情报）
- 自定义规则库

#### 2.2.3 插件化扩展
- 集成 TruffleHog、Gitleaks 等扫描器作为基准对比
- 插件化架构，用户可添加自定义扫描器

## 3. 验收标准

### 3.1 MVP 验收标准

#### 3.1.1 功能可用性

✅ **必须满足**:
- 能够扫描本地 Docker 镜像并输出 JSON 报告
- 支持 CLI 子命令（scan、history、config、verify）
- JSON 报告包含完整字段（元数据、凭证列表、统计信息）

#### 3.1.2 性能基准

✅ **必须满足**:
- 扫描速度: < 5 分钟/GB 镜像大小
- 内存占用: < 2GB（扫描中等规模镜像时）
- 支持异步处理，不阻塞主线程

#### 3.1.3 准确性目标

⚠️ **持续改进目标**（MVP 后迭代）:
- 误报率低于 TruffleHog
- 漏报率低于 TruffleHog
- 通过并行对比测试验证

### 3.2 测试验收标准

✅ **必须满足**:
- 所有单元测试通过（pytest）
- 使用当前主机上的真实镜像进行集成测试
- 测试覆盖率 > 70%

## 4. 非目标（Out of Scope）

本项目**明确不包含**以下功能：

| 功能 | 理由 |
|------|------|
| ❌ **自动修复** | 不在扫描后自动修复安全问题 |
| ❌ **漏洞扫描（CVE）** | 不进行已知的 CVE 漏洞扫描 |
| ❌ **运行时安全** | 不扫描运行中容器的内存或进程 |
| ❌ **商业化** | 作为开源项目，不提供商业 SaaS 服务 |
| ❌ **镜像构建检测** | 不监控或检测镜像构建过程 |

## 5. 用户故事

### 5.1 核心用户故事

#### 故事 1: 快速扫描本地镜像
**作为** 开发人员
**我想要** 快速扫描我刚构建的 Docker 镜像
**以便** 在推送到 Registry 前发现敏感凭证

**验收标准**:
- CLI 命令 `imagescan agent scan myapp:latest` 可执行
- 扫描结果输出到 `output/{timestamp}/{image_name}/results.json`
- 终端显示实时进度（当前扫描的层、文件）

#### 故事 2: 查看历史扫描记录
**作为** 开发人员
**我想要** 查看之前的扫描结果
**以便** 对比不同版本的镜像安全状态

**验收标准**:
- CLI 命令 `imagescan history` 列出所有历史扫描
- 支持按镜像名称、时间范围过滤
- 支持查看指定扫描的详细报告

#### 故事 3: 智能凭证识别
**作为** 安全工程师
**我想要** 系统能识别伪装或编码的凭证
**以便** 发现更隐蔽的安全风险

**验收标准**:
- LLM 能识别 Base64 编码的凭证
- LLM 能识别配置文件中的复杂凭证模式
- 提供置信度分数，标记不确定结果

#### 故事 4: 验证凭证有效性
**作为** 安全工程师
**我想要** 验证发现的凭证是否真实有效
**以便** 优先处理高风险凭证

**验收标准**:
- 静态分析凭证格式、熵值、过期时间
- 如果验证失败，提示用户手动确认
- 验证结果记录到 JSON 报告

## 6. 技术约束

### 6.1 技术选型约束

- **Python 版本**: 3.11（严格锁定）
- **LLM**: 仅使用 Google Gemini 2.5 Flash
- **数据库**: SQLite + ChromaDB（嵌入式）
- **异步框架**: asyncio

### 6.2 部署约束

- **运行边界**: 必须在容器内运行
- **部署方式**: Docker Compose / 本地安装脚本
- **配置管理**: 单一配置文件（config.toml）

## 7. 数据流

### 7.1 扫描流程数据流

```
用户输入 (镜像名称)
  ↓
主 Agent 接收任务
  ↓
主 Agent 制定计划（事件总线）
  ↓
扫描执行 Agent
  ├─ 镜像解压
  ├─ 智能文件筛选
  └─ 内容扫描
  ↓
验证分析 Agent
  ├─ 静态分析
  └─ 有效性验证
  ↓
研判反思 Agent
  └─ 置信度评估
  ↓
结果输出 (JSON + 终端)
```

### 7.2 数据存储结构

```
output/
└── {timestamp}/
    └── {image_name}/
        ├── results.json        # 扫描结果
        ├── files/              # 提取的敏感文件
        └── metadata.json       # 元数据
```

## 9. 风险与限制

### 9.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM 幻觉导致误报 | 高 | 多 Agent 协作、置信度评估、人工确认 |
| Gemini API 配额限制 | 中 | 异步处理、请求限流 |
| 大型镜像扫描耗时 | 中 | 智能文件筛选、并行处理 |

### 9.2 使用限制

- 需要有效的 Gemini API Key
- 需要访问 Docker Daemon
- 扫描结果存储在本地文件系统（不支持分布式）

## 11. 附录

### 11.1 术语表

| 术语 | 定义 |
|------|------|
| **Agent** | 基于 LLM 的智能体，可自主规划和执行任务 |
| **RAG** | Retrieval-Augmented Generation，检索增强生成 |
| **置信度** | LLM 对检测结果的可信程度（0-1 分数） |
| **误报** | 系统错误标记非敏感内容为敏感 |
| **漏报** | 系统遗漏真实的敏感凭证 |
